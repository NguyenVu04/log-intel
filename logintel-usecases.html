Thu thập & Lưu trữ · UC-01

Sidecar Thu Thập Log từ Pod

Triển khai agent Go chạy trong mọi Kubernetes pod, âm thầm thu thập log từ file và stdout/stderr, buffer và chuyển tiếp đến aggregator mà không ảnh hưởng đến ứng dụng chính.

UC-01 Sidecar Thu Thập Log từ Kubernetes Pod
Go MUST
Tiêu chí đánh giá
Điểm
Độ khó
Giá trị
Nền tảng cho toàn bộ hệ thống
P0
Cao
Tác nhân
⚙ LogIntel Sidecar Agent (Go)
👤 DevOps Engineer (cấu hình)
Điều kiện tiên quyết
  • ✓ Pod đã được inject sidecar container
  • ✓ Aggregator endpoint được cấu hình qua env var
  • ✓ Volume mount chia sẻ giữa app và sidecar
Luồng chính (Happy Path)
  1. Sidecar khởi động, đọc cấu hình từ environment variables (LOGINTEL_LOG_PATHS, LOGINTEL_AGGREGATOR_ADDR, LOGINTEL_SERVICE_NAME)
  2. Agent mở file watcher dùng fsnotify, theo dõi inode (không phải filename) để xử lý log rotation
  3. Agent attach vào Unix socket của container để capture stdout/stderr
  4. Mỗi dòng log được đọc, gán metadata (pod name, namespace, container, timestamp_ns, sequence number)
  5. Log event được đẩy vào ring buffer 64 MB in-process
  6. Background goroutine batch các event thành frame 256–512 KB và gửi qua gRPC stream đến aggregator
  7. Aggregator gửi ACK; buffer slot được giải phóng
Thiết kế quan trọng: Theo dõi bằng inode, không phải tên file. Khi log rotate xảy ra, tên file thay đổi nhưng inode mới được tạo — agent phải detect và mở file mới để không bỏ sót log.
Luồng ngoại lệ
  • Aggregator không phản hồi: exponential backoff 50ms → 30s, buffer tiếp tục nhận log. Nếu buffer đầy (> 95%), bắt đầu drop sự kiện cũ nhất và tăng counter dropped_events_total
  • Log file bị xoá trước khi đọc: ghi warning, tiếp tục watch — file mới sẽ xuất hiện khi app ghi lại
  • Sidecar crash/restart: đọc lại từ offset đã checkpoint (lưu trong emptyDir volume)
  • Cấu hình thiếu: fail fast với exit code 1 và log rõ ràng biến env nào còn thiếu
Tiêu chí chấp nhận
  • Sidecar tiêu thụ < 50m CPU và < 96 MB RAM ở trạng thái ổn định
  • Thời gian khởi động < 200ms — không được delay ứng dụng chính
  • Không mất log khi log rotation xảy ra (validated bằng integration test)
  • Counter dropped_events_total được expose qua /metrics Prometheus endpoint
  • Buffer fill % được expose như gauge — alert khi > 70% trong 60 giây liên tục
  • Mọi event đều có trường event_id (UUID v7) và timestamp_ns chính xác tới nanosecond
  • Agent khởi động lại được sau crash mà không cần can thiệp thủ công

Throughput tối thiểu50,000 dòng/giây mỗi sidecar không mất dữ liệu
Latency end-to-endDòng log → aggregator nhận được: < 500ms p99
Container image size< 20 MB (Alpine base + static Go binary)
OTLP compatibilityEvent format tương thích OpenTelemetry log record spec
Thu thập & Lưu trữ · UC-02

Chuẩn Hoá & Lưu Trữ Log Event

Rust aggregator nhận raw log events từ các sidecar, parse và validate theo canonical schema, index để full-text search, và lưu vào chunk store phân vùng theo thời gian.

UC-02 Chuẩn Hoá & Lưu Trữ Log Event
Rust MUST
Luồng chính
  1. Aggregator nhận batch LogEvent protobuf qua gRPC stream từ sidecar
  2. Validate schema: kiểm tra required fields, coerce kiểu dữ liệu, reject event có field không hợp lệ (trả về gRPC error chi tiết)
  3. Enrich metadata: geolocation nếu có IP, service version từ label map, deployment context từ K8s API
  4. Deduplication trong window 5 giây: hash (service_name + timestamp_ns + message) — drop duplicate, tăng counter
  5. Index event vào Tantivy (full-text search engine): tokenise trường message, index các label fields
  6. Ghi vào time-series chunk store: phân vùng theo 1-giờ × service_name. Chunk hiện tại (hot) ở memory, chunk cũ được flush xuống disk với nén zstd
  7. Gửi ACK về sidecar sau khi đã index thành công
Thiết kế chunk store
// Cấu trúc chunk file trên disk // Path: /data/chunks/{service}/{YYYY-MM-DD-HH}.chunk ChunkHeader { schema_version: u16, service_name: String, time_start_ns: i64, time_end_ns: i64, event_count: u64, index_offset: u64, // offset đến sparse index compression: Compression, // NONE | ZSTD } // Sparse index: mỗi 1000 events lưu 1 entry SparseIndexEntry { sequence_num: u64, timestamp_ns: i64, file_offset: u64, }
Tiêu chí chấp nhận
  • Event vi phạm schema bị reject với error message rõ ràng, không được silent drop
  • Ingest latency: raw event → searchable < 5 giây ở p99
  • Throughput: > 100,000 events/giây trên phần cứng tham chiếu (8 CPU, 32 GB RAM)
  • Full-text query trên 7 ngày log (< 500 GB) trả về trong < 2 giây
  • Chunk files có checksum để phát hiện corruption
  • Retention policy daemon xoá chunk cũ theo cấu hình RETENTION_DAYS
Yêu cầu phi chức năng
Không GC pauseRust đảm bảo zero GC trong ingest path — latency phải flat
Compression ratioCold chunks: zstd level 3, mục tiêu > 5x compression
Hot/cold splitHot = 7 ngày gần nhất trong memory-mapped files; cold = disk
Causal Intelligence · UC-03

Xây Dựng Causal Knowledge Graph

Đây là use case phức tạp nhất và tạo ra giá trị cốt lõi của LogIntel. Rust graph engine liên tục phân tích log metrics để phát hiện quan hệ nhân quả giữa các service và mã hoá chúng thành đồ thị có trọng số confidence.

UC-03 Xây Dựng Causal Knowledge Graph
Rust MUST
Tiêu chí đánh giá
Điểm
Độ khó
Giá trị
Core differentiator của LogIntel
P0
Cao
Node types
  • Service — tên, version, namespace
  • ErrorClass — fingerprint hash, frequency
  • Deployment — timestamp, services affected
  • Anomaly — type, severity, start time
  • Trace — trace_id, duration, outcome
Edge types (scored)
  • CALLS — call frequency/min
  • CAUSED — Granger p-value
  • PRECEDED — temporal proximity
  • RESOLVED_BY — resolution speed
  • SIMILAR_TO — cosine similarity
Background tasks
  • Metric extractor: 60s interval
  • Granger test: 60s interval
  • Edge decay: 1h interval
  • Graph compaction: 1h interval
  • Snapshot writer: 15min interval
Pipeline xây dựng causal edge (3 giai đoạn)
  1. Metric extraction: Với mỗi service, tính toán rolling window 60 giây: error rate (errors/s), log volume delta, warn-to-error ratio, retry log count. Chỉ chạy nếu service có ≥ 500 events trong window
  2. Granger causality test: Với mỗi cặp (A, B) có CALLS edge, chạy Granger test trên chuỗi thời gian error rate với lag 1 phút. Tạo CAUSED edge nếu p < 0.05 trong 3 window liên tiếp. Confidence = 1 - p_value
  3. Edge lifecycle: Cạnh mới: confidence = raw score. Mỗi giờ: confidence *= 0.95 nếu không được reinforce. Prune khi confidence < 0.1. Kỹ sư confirm: ghim ở confidence = 1.0, không decay
// Granger test pseudocode — Rust implementation fn granger_causality_test( series_a: &[f64], // error rate của service A series_b: &[f64], // error rate của service B lag: usize, // = 1 minute ) -> GrangerResult { // Unrestricted model: B(t) ~ B(t-1..lag) + A(t-1..lag) let rss_unrestricted = fit_var_model(series_a, series_b, lag).residual_ss; // Restricted model: B(t) ~ B(t-1..lag) only let rss_restricted = fit_ar_model(series_b, lag).residual_ss; let f_stat = compute_f_statistic(rss_restricted, rss_unrestricted, lag, n); GrangerResult { p_value: p_from_f(f_stat, lag, n), f_statistic: f_stat } } // Tạo CAUSED edge nếu đủ điều kiện if result.p_value < 0.05 && consecutive_windows >= 3 { graph.add_caused_edge(service_a, service_b, 1.0 - result.p_value); }
Tiêu chí chấp nhận
  • CALLS edges xuất hiện trong vòng 5 phút sau khi có đủ log data
  • CAUSED edges chỉ được tạo khi p < 0.05 trong 3 window liên tiếp (không tạo từ 1 window đơn lẻ)
  • Service có < 500 events/window không được đưa vào Granger test
  • Precision của CAUSED edges: > 70% được kỹ sư xác nhận khi được prompt (đo sau 30 ngày)
  • Graph trong memory sử dụng petgraph::StableGraph — node index không thay đổi sau khi xoá node
  • Snapshot 15 phút/lần được lưu và queryable theo timestamp
Rủi ro thiết kế: False positive CAUSED edges dẫn kỹ sư đi sai hướng trong sự cố — nguy hiểm hơn việc không có edge. Ưu tiên precision hơn recall. Luôn hiển thị confidence interval trong UI.
Runbook Learning · UC-06

Ghi Nhận Giải Quyết Sự Cố

Khi kỹ sư đánh dấu sự cố đã giải quyết qua lệnh /resolve trong chat, hệ thống tự động tổng hợp toàn bộ cuộc trò chuyện thành một runbook entry có cấu trúc và lưu vào runbook store.

UC-06 Ghi Nhận Giải Quyết Sự Cố → Runbook Store
Java MUST
Trigger
  • Kỹ sư gõ /resolve trong chat interface
  • Hoặc click nút "Mark as Resolved" trên incident panel
Tác nhân
👤 On-call Engineer
⚙ Runbook Learning Service (Java)
Luồng chính
  1. Engineer gõ /resolve — chat service gửi event INCIDENT_RESOLVED đến Runbook Learning Service với session_id và timestamp
  2. Service truy xuất toàn bộ conversation history của session từ chat store
  3. LLM được prompt để tổng hợp: trích xuất symptom description, action steps đã thực hiện theo thứ tự, expected outcome
  4. Tạo symptom fingerprint: embed câu mô tả sự cố bằng all-MiniLM-L6-v2 (ONNX, 22 MB) → vector 768 chiều
  5. Truy vấn causal graph tại thời điểm sự cố: lấy subgraph 2 hop xung quanh các service bị ảnh hưởng
  6. Tạo structural fingerprint từ subgraph: hash topology (node types + edge types, không phải tên service cụ thể)
  7. Lưu RunbookEntry vào PostgreSQL với confidence khởi đầu = 0.5 (pending validation)
  8. Hệ thống bắt đầu monitoring 30 phút tiếp theo: theo dõi error rate của các service liên quan
  9. Sau 30 phút: nếu error rate trở về baseline → confidence tăng lên 0.8 (confirmed). Nếu vẫn cao → giữ nguyên 0.5, gắn flag cần review
// RunbookEntry schema (Java record) record RunbookEntry( String runbookId, // rb_{uuid_short} Instant createdAt, Instant lastTriggered, int triggerCount, double confidence, // 0.0 – 1.0 SymptomFingerprint fingerprint, List<ResolutionStep> steps, long expectedResolutionSecs, List<String> postResolutionSignals, // điều kiện để confirm resolved boolean driftFlag, Instant driftDetectedAt ) {} record SymptomFingerprint( List<String> errorClasses, List<String> affectedServices, String graphSubgraphHash, float[] embeddingVector // 768-dim từ all-MiniLM-L6-v2 ) {}
Tiêu chí chấp nhận
  • Lệnh /resolve nhận response trong < 3 giây
  • Runbook entry luôn có ít nhất 1 resolution step được trích xuất
  • Embedding vector được lưu chính xác (validate bằng cosine similarity test với chính nó = 1.0)
  • 30-minute validation window chạy async, không block chat
  • Kỹ sư có thể chỉnh sửa runbook entry trong vòng 1 giờ sau khi tạo
  • Runbook với confidence < 0.4 sau validation được đánh dấu "low-confidence" và không được auto-suggest
Runbook Learning · UC-07

Đề Xuất Runbook Khi Có Sự Cố

Khi hệ thống phát hiện sự cố mới, tự động tìm kiếm runbook tương tự trong store và đề xuất cho kỹ sư inline trong chat với confidence score và thời gian giải quyết ước tính.

UC-07 Đề Xuất Runbook Phù Hợp Khi Có Sự Cố Mới
Java MUST
Pipeline matching 2 giai đoạn
  1. Coarse filter (Jaccard similarity): So sánh tập error_classes và affected_services giữa sự cố hiện tại và tất cả runbook. Giữ lại các runbook có Jaccard(error_classes) + Jaccard(services) > 0.6. Kết quả: thường còn 5–20 candidates
  2. Fine ranking (embedding cosine similarity): Embed context sự cố hiện tại → so sánh cosine similarity với embeddingVector của các candidates. Sắp xếp giảm dần theo similarity score
  3. Filter bởi confidence: Chỉ đề xuất runbook có confidence ≥ 0.6 và không có drift_flag = true
  4. Render top-3: Trả về tối đa 3 runbook dưới dạng structured JSON để React render thành expandable card inline trong chat
Format response đề xuất
// Response từ Runbook Suggestion API { "suggestions": [ { "runbook_id": "rb_a3f9c2", "title": "Redis memory pressure → payment-svc timeout", "confidence": 0.91, "similarity_score": 0.87, "trigger_count": 7, "expected_resolution_secs": 180, "steps": [ /* ResolutionStep[] */ ], "drift_flag": false } ], "incident_context": { "affected_services": ["payment-svc", "redis-cache"], "causal_path": "payment-svc ←[0.91]← redis-cache" } }
Tiêu chí chấp nhận
  • Đề xuất xuất hiện trong chat < 2 giây sau khi sự cố được detect
  • Acceptance rate (kỹ sư click "Apply") > 40% sau 90 ngày sử dụng
  • Runbook có drift_flag = true tuyệt đối không được đề xuất
  • Khi không có runbook nào đủ điều kiện, hiển thị "No matching runbooks — be the first to resolve and save this pattern"
  • Mỗi đề xuất có nút "This didn't work" để kỹ sư reject và trigger drift detection
UX quan trọng: Đề xuất runbook phải xuất hiện proactively khi sự cố được phát hiện, không chỉ khi kỹ sư hỏi. Kỹ sư lúc 3 giờ sáng không nên phải nhớ câu lệnh gì để hỏi — hệ thống phải chủ động.
NL Query Compiler · UC-09

Biên Dịch Câu Hỏi Tự Nhiên → LogQL+

Người dùng đặt câu hỏi bằng tiếng Anh tự nhiên. Hệ thống biên dịch thành LogQL+ AST có kiểu dữ liệu, hiển thị query đã biên dịch để người dùng xác nhận, sau đó thực thi an toàn.

UC-09 Biên Dịch Câu Hỏi Tự Nhiên → Typed LogQL+ AST
Java MUST
Pipeline biên dịch 4 bước
  1. Intent classification: Fine-tuned classifier (LangChain4j) phân loại query thành một trong: LOG_SEARCH, GRAPH_QUERY, RUNBOOK_LOOKUP, ANOMALY_QUERY, AGGREGATION. Confidence < 0.7 → hỏi lại người dùng
  2. Entity extraction (NER): ONNX NER model trích xuất: service names (so khớp với service registry), time ranges (parse relative time "yesterday", "last 2h"), severity levels, error class names, metric thresholds
  3. AST synthesis: LLM được cung cấp intent + entities → tạo JSON AST có kiểu. Prompt bao gồm LogQL+ grammar spec và ví dụ few-shot. LLM không bao giờ tạo raw string
  4. Validation: AST được validate bởi Java schema validator. Lỗi được trả về LLM để sửa (tối đa 2 lần retry). Nếu vẫn lỗi → trả về error message cho người dùng với gợi ý reformulate
LogQL+ AST schema
// Typed AST — không bao giờ là raw string record LogQuery( QueryType type, List<Filter> filters, Optional<TimeRange> timeRange, Optional<Aggregation> aggregation, Optional<GraphTraversal> graphOp, // chỉ cho GRAPH_QUERY Optional<Sort> sort, int limit ) {} record Filter(String field, FilterOp op, String value) {} // FilterOp: EQ | NEQ | CONTAINS | GTE | LTE | IN | REGEX record GraphTraversal( String fromService, EdgeType traverseEdge, // CAUSED | CALLS | RESOLVED_BY double minConfidence, int maxDepth ) {}
Ví dụ biên dịch thực tế
// Input: "Show timeout errors in payment-svc in the last 2 hours" // Output AST: { "type": "LOG_SEARCH", "filters": [ { "field": "service_name", "op": "EQ", "value": "payment-svc" }, { "field": "severity", "op": "GTE", "value": "ERROR" }, { "field": "message", "op": "CONTAINS", "value": "timeout" } ], "timeRange": { "relative": "2h" }, "sort": { "field": "timestamp_ns", "direction": "DESC" }, "limit": 100 }
Tiêu chí chấp nhận
  • Compilation success rate > 85% (query compile thành công không cần người dùng reformulate)
  • Compiled query luôn được hiển thị cho người dùng trước khi thực thi — không execute ngầm
  • Thời gian biên dịch < 3 giây từ lúc người dùng submit đến khi query được hiển thị
  • LLM không được phép tạo raw LogQL+ string — validate bằng type checking trước khi serialize
  • Mọi compilation attempt (kể cả thất bại) được log để cải thiện model
  • Người dùng có thể chỉnh sửa LogQL+ AST trực tiếp sau khi xem compiled query
Security: AST-based compilation ngăn chặn prompt injection. Nếu người dùng cố nhúng SQL injection hay LogQL injection vào câu hỏi, nó sẽ bị bắt tại bước schema validation trước khi đến execution layer.
Giao diện React · UC-11

Xem Log & Live Tail

Log explorer hiển thị log events với virtual scrolling, hỗ trợ live tail qua SSE, filter đa chiều, và highlight các dòng log tham gia vào causal edge đang active.

UC-11 Log Explorer — Xem Log & Live Tail
React MUST
Tính năng core
  • TanStack Virtual: tối đa 200 DOM rows, cuộn mượt với 1M+ dòng log
  • Live tail via SSE với nút Pause/Resume
  • Filter chips: service, severity, time range, label key=value
  • Filter state encode vào URL — link shareable
  • Highlight log line tham gia causal edge (màu theo confidence)
Tương tác nâng cao
  • Click log line → expand detail panel
  • Nút "Explain this error" → mở chat với context
  • Log line có trace_id → expand reconstructed trace
  • Copy log line, copy as LogQL+ filter
  • Keyboard shortcuts: j/k điều hướng, / để search
Tiêu chí chấp nhận
  • Render 200 visible rows với frame rate ổn định ≥ 60fps
  • Live tail latency (server emit → visible trong UI) < 1 giây
  • Filter thay đổi → kết quả cập nhật < 500ms
  • URL với filter state có thể chia sẻ và reproduce chính xác
  • Không bao giờ dùng innerHTML để render log content (XSS prevention)

ReactRustSHOULD
